# 产品评审需求单撰写（整合结构化的轮询内容）
---

## Core Rules — 速查

1. **禁止空话套话。** 不使用"打造极致体验""行业领先"等空洞无实际意义的词。
2. **数据优先。** 有数据写数据，无数据一定要标注「待补充」，不自行编造和杜撰。
3. **边界思维。** 需求详情产出后扫描所有边界情况，至少覆盖 5 种边界维度，每种维度下拆分子场景。
4. **中文引号。** 使用「」而非 ""。
5. **禁止 emoji。** 正文中不使用任何 emoji。
6. **版本记录。** 每次生成/修改需求单，更新版本记录表，版本记录表放在文档头部。

---

## Process Flow

```
收到输入（结构化笔记 或 用户直接描述）
  │
  ├─ ① 按标准模板撰写（10 节 + 版本记录前置）
  │    ├─ 用户角色合并入需求背景，不设单独章节
  │    ├─ 核心主流程需要附 ASCII 流程图
  │    ├─ 非功能需求标注数据来源
  │    └─ 边界场景 ≥ 5 个维度
  │
  ├─ ② 应用 7 条写作规则
  │
  ├─ ③ 逐项通过验证清单
  │
  └─ ④ 输出 → .md 文件 → present_files
```

## ③ Writing Rules — 7 条写作红线

| # | 规则 | 说明 |
|---|------|------|
| 1 | 禁止空话套话 | 不使用"打造极致体验""行业领先""颠覆性"等空洞词 |
| 2 | 数据优先 | 有数据写数据并标注来源；无数据标注「待补充」「待与 X 团队对齐」 |
| 3 | 边界思维 | 至少覆盖 5 种边界维度，每种维度拆分子场景。参考 `~/.workbuddy/skills/bubble-prd-interview/reference2.md` 中第 5 节 Case 关于边界的处理方式 |
| 4 | 中文引号 | 正文中使用「」而非 "" |
| 5 | 禁止 emoji | 正文中不使用任何 emoji |
| 6 | 版本记录 | 版本记录放在文档头部，每次更新追加新行 |
| 7 | 非功能需求标注 | ① 每项非功能需求先写定性要求（如「语音播放体验流畅」「所有文件须加密存储」）；② 标注定量指标的数据来源状态（✅有来源 / ⏳待对齐 / ❓待补充）；③ 完全无法获得数据则只写定性要求，不编造 |
